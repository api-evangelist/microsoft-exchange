<p>We’re excited to announce the public preview of <a href="https://aka.ms/NAAdocs">Nested App Authentication (NAA)</a>. NAA provides simpler authentication and top tier identity protection through APIs designed specifically for add-ins in Office hosts.</p>
<p>We’re also announcing that <strong>legacy Exchange <a href="https://learn.microsoft.com/en-us/office/dev/add-ins/outlook/authentication#exchange-user-identity-token">user identity tokens</a> and </strong><a href="https://learn.microsoft.com/office/dev/add-ins/outlook/authentication#callback-tokens"><strong>callback tokens</strong></a><strong> will be turned off by default for all Exchange Online tenants in October 2024.</strong> This is part of <a href="https://blogs.microsoft.com/on-the-issues/2023/11/02/secure-future-initiative-sfi-cybersecurity-cyberattacks/">Microsoft’s Secure Future Initiative</a> to give organizations the tools they need in the current threat landscape. Add-in developers who access Exchange data through EWS or Outlook REST must take immediate action to ensure their add-ins are ready before legacy Exchange tokens are off by default in October 2024.<a id="post-20361-_Hlk160439908"></a> NAA is the best authentication option for affected add-ins; we recommend beginning work on proof of concepts using the NAA preview and adopting NAA soon after general availability.</p>
<h2>The email threat landscape</h2>
<p>Last year, Microsoft published data showing the <a href="https://www.microsoft.com/security/blog/2023/05/24/volt-typhoon-targets-us-critical-infrastructure-with-living-off-the-land-techniques/">increasing frequency and sophistication</a> of attacks by brazen nation state actors. We also documented our approach to protecting customers from these attacks: <a href="https://blogs.microsoft.com/on-the-issues/2023/11/02/secure-future-initiative-sfi-cybersecurity-cyberattacks/">Microsoft’s </a><a href="https://blogs.microsoft.com/on-the-issues/2023/11/02/secure-future-initiative-sfi-cybersecurity-cyberattacks/">Secure Future Initiative</a>.</p>
<p>A core pillar of this initiative is <a href="https://www.microsoft.com/en-us/security/blog/2023/11/02/announcing-microsoft-secure-future-initiative-to-advance-security-engineering/"><em>strengthening identity protection against highly sophisticated attacks</em></a>. Email data is valuable and constantly pursued by bad actors; Microsoft and our add-in partners must strengthen add-in identity protection. Add-in developers know they should rely on proven authentication libraries, such as Microsoft’s MSAL.js, to handle the vagaries of authentication while developers focus on their apps’ unique value. However, relying on libraries in the Office add-in space was difficult: add-ins run within a host app, which makes it challenging to get user consent to access resources on a user’s behalf, accept authentication factors from users, and operate within the Office environment.</p>
<p>NAA simplifies Office add-in specific authentication with APIs that work for add-ins nested within Office hosts, making it simple to get consent, accept the latest and safest authentication factors, and allow customer admins to secure their environment with Entra ID policies.</p>
<p>Alongside our announcement of NAA, we’re preparing to deactivate legacy Exchange <a href="https://learn.microsoft.com/office/dev/add-ins/outlook/authentication#exchange-user-identity-token">user identity tokens</a> and <a href="https://learn.microsoft.com/office/dev/add-ins/outlook/authentication#callback-tokens">callback tokens</a> for all Exchange Online tenants in October 2024. Unlike the Entra ID tokens provided through NAA, these legacy tokens no longer provide sufficient support for organizations’ response to threats against email data.</p>
<h2>Summary of timeline and changes</h2>
<ul>
<li>April 2024
<ul>
<li>NAA enters public preview. Any add-in using legacy Exchange <a href="https://learn.microsoft.com/office/dev/add-ins/outlook/authentication#exchange-user-identity-token">user identity tokens</a> and <a href="https://learn.microsoft.com/office/dev/add-ins/outlook/authentication#callback-tokens">callback tokens</a> must be migrated to NAA. Add-in developers should evaluate which of their add-ins use legacy Exchange tokens and begin planning and development to migrate all affected add-ins.</li>
</ul>
</li>
<li>October 2024
<ul>
<li>Exchange Online blocks legacy Exchange <a href="https://learn.microsoft.com/office/dev/add-ins/outlook/authentication#exchange-user-identity-token">user identity tokens</a> and <a href="https://learn.microsoft.com/office/dev/add-ins/outlook/authentication#callback-tokens">callback tokens</a> in all tenants by default. Add-ins that haven’t adopted NAA and rely on legacy Exchange tokens will be unable to call EWS and Outlook REST unless admins opt into continued legacy token issuance.</li>
<li>Exchange 2019 and other on-premises versions of Exchange won’t block legacy Exchange <a href="https://learn.microsoft.com/office/dev/add-ins/outlook/authentication#exchange-user-identity-token">user identity tokens</a> and <a href="https://learn.microsoft.com/office/dev/add-ins/outlook/authentication#callback-tokens">callback tokens</a>.</li>
</ul>
</li>
</ul>
<h2>Adopt NAA in add-ins as soon as possible</h2>
<h3>Approaches for new add-ins</h3>
<p>NAA is the best approach for any new add-in development. You may also use other approaches for authentication such as the <a href="https://learn.microsoft.com/office/dev/add-ins/develop/sso-in-office-add-ins">on-behalf-of flow single sign-on (SSO) getAccessToken API in office.js</a>, custom implementations of MSAL.js, or <a href="https://learn.microsoft.com/office/dev/add-ins/outlook/authentication#single-sign-on-access-token">Entra ID SSO</a>. Exchange legacy tokens and EWS or Outlook REST solutions aren’t permitted or supported.</p>
<p>To get access to Exchange resources, you should use Microsoft Graph. Graph is the <a href="https://devblogs.microsoft.com/microsoft365dev/retirement-of-exchange-web-services-in-exchange-online/">best approach for access to Exchange Online resources—EWS is on the path to retirement</a>. Exchange Online will stop issuing legacy Exchange <a href="https://learn.microsoft.com/office/dev/add-ins/outlook/authentication#exchange-user-identity-token">user identity tokens</a> and <a href="https://learn.microsoft.com/office/dev/add-ins/outlook/authentication#callback-tokens">callback tokens</a> in October 2024.</p>
<h3>Approaches for existing add-ins</h3>
<p>For existing add-ins, developers need to evaluate whether their add-ins use legacy Exchange <a href="https://learn.microsoft.com/office/dev/add-ins/outlook/authentication#exchange-user-identity-token">user identity tokens</a> and <a href="https://learn.microsoft.com/office/dev/add-ins/outlook/authentication#callback-tokens">callback tokens</a> as soon as possible. These tokens are used by add-ins to request resources from EWS and Outlook REST, which are on <a href="https://devblogs.microsoft.com/microsoft365dev/retirement-of-exchange-web-services-in-exchange-online/">deprecation pathways</a>. To evaluate whether your add-in uses legacy Exchange tokens, look for calls to the following APIs.</p>
<ul>
<li><a href="https://learn.microsoft.com/javascript/api/outlook/office.mailbox?view=outlook-js-preview#outlook-office-mailbox-makeewsrequestasync-member(1)">makeEwsRequestAsync</a></li>
<li><a href="https://learn.microsoft.com/javascript/api/outlook/office.mailbox?view=outlook-js-preview#outlook-office-mailbox-getuseridentitytokenasync-member(1)">getUserIdentityTokenAsync</a></li>
<li><a href="https://learn.microsoft.com/javascript/api/outlook/office.mailbox?view=outlook-js-preview#outlook-office-mailbox-getuseridentitytokenasync-member(1)">getCallbackTokenAsync</a></li>
</ul>
<p>If your add-in uses legacy Exchange tokens and works with Exchange Online, you must adopt NAA as soon as possible. This is necessary to maintain compatibility with Exchange Online. Add-ins used only in on-premises environments don’t need to be updated.</p>
<p>We also recommend adopting NAA if you use <a href="https://learn.microsoft.com/office/dev/add-ins/develop/sso-in-office-add-ins">Office SSO</a> today. The Office SSO API isn’t deprecated, but NAA is easier to implement as a developer and provides a better user experience.</p>
<h3>Technical steps to adopt NAA</h3>
<p>To adopt NAA in each of your add-ins, you should follow these steps:</p>
<ol>
<li>Register your add-in with Entra ID as an application.</li>
<li>Update your redirect URIs to support trusted brokers.</li>
<li>Update your MSAL.js configuration to allow native bridging.</li>
<li>Add a fall back authentication method.</li>
<li>Test your add-in.</li>
</ol>
<p>The steps to support NAA are covered here: <a href="https://aka.ms/NAAdocs">Enable SSO in an Office Add-in using nested app authentication (preview).</a></p>
<h2>Simpler and safer authentication for Outlook add-ins</h2>
<p>The launch of NAA provides Outlook add-in developers with a simpler and safer way to do add-in authentication. At the same time, the end of support for legacy Exchange tokens helps organizations secure their identity and access management estate. The work that each add-in developer does to migrate away from legacy Exchange tokens and EWS/Outlook REST helps organizations across the world keep the vital data in their Exchange environments safe.</p>
<p>We recognize that the October 2024 end of support for legacy Exchange tokens will come quickly. We recommend developers immediately start to identify add-ins that use legacy tokens using the steps described in this blog post and in our documentation. If your add-in consumes legacy tokens, you should make plans to migrate to Entra ID tokens and Microsoft Graph before October 2024. Your investment in this migration will pay dividends for organizations that can better manage identity and access in add-ins moving forward.</p>
